## 本版主要变更（v2026.06.10）

本次以"硬件协议对齐参考程序 + 设备测试 + 设备状态可视化"为主线，覆盖主服务器、HTML race_control.html、EXE RemoteTimingControl 三端。

### 1. 硬件计时器协议改为参考程序格式

参考 `C:\2026年11月2日PM SWIM串口通讯程序Server`（CSetMatchTime / SwimView.cpp）。原先用 `0x42 + 子码` 逐项发送的方式硬件不识别，现一律改成参考的命令编号 + 单帧打包：

| 命令 | 含义 | 帧字节布局 |
|---|---|---|
| `0x41` Set_ArmDelay_Time | 触板/盲表/出发台 一帧打包 | d3 ArmDelayNormal · d4 ArmDelayAfterStart · d5 MBDelay · d6 ResultDisplay · d7 EdgeBit · d8 LaneDir<<4\|SP50M<<1\|SP · d9 TPDelay · d10 SBDelay |
| `0x44` Set_PoolConfiguration_Com1 | 泳池参数 | d3 泳道数 · d4 泳池长度 |
| `0x45` Set_MB_Num | 盲表数量 | d3 左端 · d4 右端 |
| `0x46` Set_TPSBMB_State | 触板/出发台/盲表1 启用位（一帧含 TP/SB/MB 全 20 位）| 8 字节位流：State[0..2] TP · State[3..4] SB · State[5..6] MB1 · State[7] 高/低半字节=SB16-19/MB16-19 |
| `0x47` Set_StartPlaceLaneDir | 发令位置 + 道次方向（位）| 与 0x41 d8 对齐 |
| `0x49` / `0x4A` 扩展 | 盲表 2 / 盲表 3 启用位 | 与 0x46 MB 段同 20 位编排（3 字节）|

> 0x49 / 0x4A 是为我们每端 3 个盲表设计的扩展。当前固件未识别这两条会忽略；待硬件升级后无需改本程序，把协议告诉硬件方即可。
>
> `TimingBridge.SendFullFrame` 已扩展到 d9/d10（之前固定为 0），并把 hex 日志打全 8 个数据字节，便于联调时双方核对。

### 2. 设备五态："已触板"红色

`DeviceStatus` 枚举增加 `Touched` 状态，颜色红 `#EF4444`。运动员触板（或盲表/出发台）出正式成绩后，对应那一端的设备点先转 Touched 红，到点再转 Closed。窗口期内的额外触发：

| 设备 | Touched 保持时间 | 期间事件归属 |
|---|---|---|
| 触板 | `ResultConfirmCloseDelay`（默认 3s）| 写日志 + 加入 `LaneResult.BackupTouchTimes` |
| 盲表（按单只）| `ResultConfirmCloseDelay` | 写日志 + 加入 `LaneResult.BackupBlindTimes`，格式 `盲号:时间` |
| 出发台 | `StartBlockCloseDelay`（默认 3s）| 写日志 + 加入 `LaneResult.BackupReactionTimes`，格式 `端:时间` |

为避免与"已触板红"冲突，**Broken 改为黑色 `#000000`**（白色细描边在深底上保留可见性）。配色：

| 状态 | 颜色 |
|---|---|
| Open 接收中 | 🟢 绿 `#22C55E` |
| **Broken 损坏** | ⚫ **黑 `#000000`** |
| Touched 已触板 | 🔴 红 `#EF4444` |
| FalseStart 抢跳 | 🟡 黄 `#F59E0B` |
| Closed 关闭 | ⬜ 灰 `#475569` |
| NotInstalled 未安装 | 虚线框 |

### 3. 备用成绩展示

进入"查询比赛原始数据"窗口（`Database/RawData` 中的 `.txt` / `.html` 文件）每位运动员的"分段明细"行下方，**Touched 窗口期内捕获的额外计时事件**作为"备用成绩（争议时使用）"列出：

```
★备用成绩(争议时使用):  触板[26.45]  盲表[1:26.51, 2:26.49]  出发台[left:0.085]
```

HTML 版本用红底加粗突出。三种数据都为空时整行不出现。

### 4. 道次正序 / 逆序

参数设置增"**道次顺序**"单选：正序 `0→9`（顶到底）/ 逆序 `9→0`。同步项：

- 写入 `timing_settings.json`，重启自动恢复
- 0x41 帧 d8 位 4（`LaneDirectBit`）下发硬件
- 主服务器、HTML、EXE 三端泳道面板按选定顺序排列；广播 `laneCloseSettings.laneOrder`

### 5. 设备状态持久化

新增 `device_states.json`（同 EXE 目录），持久化以下用户配置类字段（运行状态不进文件）：

- 触板/盲表 1-3/出发台 `*Broken` × 左/右
- 盲表 1-3 `*NotInstalled` × 左/右
- `LeftManualEnabled` / `RightManualEnabled`

保存触发：「设备状态管理」/「手动按键管理」对话框确定后，以及任何远端 `SET_DEVICE_STATUS` / `SET_MANUAL_STATUS` WebSocket 命令。加载时机：每次 `InitLaneDeviceStates()` 之后跟随 `ApplyPersistedDeviceStates()`（启动、加载赛事档案、改泳道数 3 处）。

> **重要 bug 修复**：`LoadLastCompetition` 里曾经多调一次 `InitLaneDeviceStates()`，把启动时刚加载的设备状态冲掉，造成"上次设的损坏标志看不到"。已修。

### 7. 系统硬件 栏 + 设备测试 模式

主服务器比赛控制右栏新增"**系统硬件**"区块，含两颗按钮（HTML/EXE 顶部 nav 也对应放上）：

- **连接串口**：用 `timing_connection.json` 里保存的串口/波特率快速连接；已连接时变红"断开"
- **设备测试**：参考程序 `OnBnClickedTestSystemButton` 的等价功能

设备测试行为：
1. 弹窗确认 → 进入 `_testMode`
2. 清空 `_rawTimingLog`、滚动时间归零
3. 所有非损坏/非未安装设备置 Open
4. 发 `0x1D Test_Command` 给硬件
5. 硬件来事件 → 跳过比赛状态/空泳道过滤 → 写日志 `【测试】泳道X 类型 时间`
6. 该端设备点 **红闪 500ms 后回绿**，姓名栏显示左端事件文字（"触 23.45"/"出发 0.42"），成绩栏显示右端事件文字
7. 三端（服务器+HTML+EXE）状态徽章覆盖红色"设备测试"
8. 测试期间不创建 split / 不计成绩 / 不进 Touched 窗口
9. 再点一次按钮退出，所有设备 Closed

HTML/EXE 通过 WebSocket `DEVICE_TEST_TOGGLE` / `QUICK_CONNECT_SERIAL` 命令请求，服务器统一切换。`HandleTimingCommand` 入口为所有远端命令写日志便于排查。

## 1. 系统概述

### 1.1 设计目标

本系统是一套**完整的游泳比赛管理与计时解决方案**，覆盖赛事全生命周期：

- **赛前**：赛事创建 / 运动员报名 / 自动或手动分组 / 秩序册编辑 / 纪录数据导入
- **赛中**：硬件计时实时联动 / 三端控制台同步 / 设备状态管理 / 大屏展示 / 媒体播放
- **赛后**：成绩确认与排名 / 多轮晋级 / 成绩册编辑 / 多格式文档输出（PDF / DOC / HTML）

系统遵循 **World Aquatics（FINA）SW 3 竞赛规则** 自动分组（循环 / 直接排位 + 矛头泳道分配）。

### 1.2 设计原则：服务器是单一数据源

所有大屏、排名屏、文档（编辑 / 输出 / 打印）显示的数据**都来自比赛控制端（主服务器）**。前端浏览器/打印模板**直接渲染服务器广播的字段**（`sw.rank`、`sw.reactionTime`、`sw.legReactionTimes`、`sw.recordNote` 等），**不再做二次重排或重算**。

> 这条原则是 v2026.05.20 起的硬约束：任何"大屏显示某字段不对"、"打印某列不对"的问题，都先回到主服务器的记录/计算路径去修，让广播本身的数据正确，再让前端原样渲染。前端纯展示层格式化（小数位、HTML 转义、奖牌色映射）仍可在前端做。

## 3. 系统架构与通讯

```
            ┌──────────────────────────────┐
            │        计时硬件控制器         │
            │   (串口 / UDP / TCP)         │
            └──────────┬───────────────────┘
                       │ 12 字节帧 F1 53 ... F4
                       │（双向同步：参数 / 设备 / 比赛距离）
        ┌──────────────▼──────────────────┐
        │   主服务器 SwimmingScoreboard    │
        │   (WebSocket :3002 / JSON 持久化) │
        └──────────────┬──────────────────┘
                       │
        ┌──────────────┼─────────────────┬──────────────┐
        ▼              ▼                 ▼              ▼
   EXE 远程台    HTML 计时台      大屏 / 排名屏     在线报名 / 检录台
```

### 3.1 三个计时控制台等价

主服务器 / `RemoteTimingControl.exe` / `race_control.html` **功能完全等价**——任何一端按下就位 / 发令 / 确认成绩 / 计时器清零，都通过 WebSocket 广播到其它两端 + 硬件，状态保持一致。

### 3.2 数据流单向：服务器 → 客户端

- **运动员姓名/代表队/反应时/分段/名次/纪录** 全部由服务器在广播载荷中直接给出。
- 客户端不再二次重排名次、不再前端 `\n / 字符串拼接` 推算反应时。如发现某字段显示异常，先在主服务器修，让广播数据本身正确。

### 3.3 硬件计时协议（摘要）

- 12 字节帧 `F1 53 [子码] [数据 ×8] F4`
- `0x42` 子码扩展（双向）：
  - `0x01-0x07`：泳道关闭时间 / 出发台延迟 / 抢跳阈值 / 终点位置
  - `0x08`：左右盲表数量（高 4 位左 / 低 4 位右）
  - `0x10`：设备好/坏位图
  - `0x20`：比赛距离 + 左右触板次数
- 详见 `游泳计时系统通讯协议.pdf`

## 8. 赛程与分组

### 8.1 一键生成日程

点击 **一键生成日程**，弹窗让用户为所有项目选择赛次方案：

- 只决赛（推荐，适合中小型赛事）
- 预赛 → 决赛
- 预赛 → 半决赛 → 决赛

系统按"组别 → 性别 → 项目 → 赛次"维度合成场次，按距离 / 项目类型自动排序到上午（预赛）/ 下午（半决赛）/ 晚上（决赛）三段：

| 时段 | 起始时间 |
|---|---|
| 上午 | 09:00 |
| 下午 | 14:00 |
| 晚上 | **17:30**（v2026.04 由 19:00 改为 17:30）|

### 8.2 预赛自动分组（FINA SW 3 规则）

按 **World Aquatics SW 3.1.1.4 / 3.1.1.5 / 3.1.2** 标准实现：

#### 短距离项目（50 / 100 / 200 米）

- `seedHeats = min(总组数, 3)`：**最后 3 组**进行循环（种子）排位
- 前面组进行直接排位，由慢到快编号；最慢组（第 1 组）拿剩余不满人数
- 种子分配：第1名→最后组中道，第2名→倒数第2组中道，第3名→倒数第3组中道；第4名→最后组次中道（lanePriority[1]），依次循环填满

#### 中长距离 / 接力（400 / 800 / 1500 米 / 4×N 接力）

- `seedHeats = min(总组数, 2)`：**最后 2 组**进行循环（种子）排位
- 前面组直接排位

> 实现位置：`HeatScheduler.AssignFinaSeeding(...)`（个人 `GenerateHeats` 与接力 `GenerateRelayHeats` 共用），按 `IsShortDistanceEvent(eventName)` 自动判定距离类别。

#### 矛头泳道（Spearhead）分配

| 泳池 | 优先级（最快→最慢）|
|---|---|
| 8 道 | `4, 5, 3, 6, 2, 7, 1, 8` |
| 10 道 | `4, 5, 3, 6, 2, 7, 1, 8, 0, 9` |

成绩最快者必占 4 道；最慢者排到边道（10 道场景下 0/9 道）。

> 注：8 人以下单组直接决赛时，0、9 道空着不用，FINA 标准如此。

### 8.3 出场编排微调

`出场编排微调` Tab 的 DataGrid 可对每位运动员的"组次/道次"做精细调整。支持：
- **手动均分**：输入"每组 N 人"自动分组
- **同组互换泳道**
- **追加分组**：把新报名的运动员塞到最后一组空位 + 必要时新建组

### 8.4 晋级处理

`成绩与排名` Tab 的 **晋级处理** 按钮 → 弹出 `PromotionQueryWindow`：

- 查询当前赛次的总排名（不分组统一排名）
- 设置晋级人数（默认：预赛→半决赛取 16，半决赛→决赛取 8）
- 一键执行晋级 + 自动分组到下一赛次

每组确认成绩后系统也会**主动询问**是否进入晋级流程（仅当全部组完赛）。

### 9.3 泳道实时状态

每条泳道一行，左右对称布局：

```
[T] 盲3 盲2 盲1 出发 触板 [圈 ▲▼] │ 姓名+进度 │ [▲▼ 圈] 触板 出发 盲1 盲2 盲3 [T]
```

- **左右盲表数量**：可独立设置 1-3，超出数量的圆点占位但隐藏（盲1 / 出发 / 触板的位置始终固定）
- **T 按钮**：手动触板，左右各一，可独立启用 / 禁用
- **进度条**：箭头方向跟随发令位置；显示运动员所在段
- **接力项目**：第 1 行显示代表队（如"江苏"），第 2 行显示 `第N棒: 队员姓名`（按当前圈数自动推断）
- **空泳道**：整行淡化（透明度 35%），名字显示"（空泳道）"
- **圈数 ▲▼ 微调**：每道左 / 右各有一对 spinner，按哪一侧只调整哪一侧的剩余圈数显示。值钳到 `[0, 该侧实际触板次数]`：50m 出发端 0 / 到达端 1；100m 双侧各 1；150m 出发端 1 / 到达端 2；200m 双侧各 2；以此类推。三端（主控 / EXE / HTML）操作一致，结果由服务器统一计算后实时广播。
- **接力棒次随圈数即时切换**（v2026.05.30）：服务器据 spinner 偏移后的"显示用当前圈数"`displayedCurrentLap = 总圈数 − 左剩余 − 右剩余` 推算"第N棒"，并写入广播。大屏直播视图的右列、HTML 计时控制台、EXE 远程台的接力姓名都用此值；按 ▼ 加圈即时进入下一棒，按 ▲ 减圈回退一棒。
- **反应时颜色**：硬件检测到反应时低于阈值时，反应时数字变 **红色** 提醒裁判；行边框琥珀色提示"可疑起跳"。**不再自动 DSQ**——是否判罚由裁判手动按 DSQ 按钮决定。

### 9.4 比赛控制按钮

- **就位** → 锁定泳道、出发台打开
- **发令** → 启动计时器、关闭出发台
- **确认本组成绩** → 锁定本组成绩并永久入档
- **计时器清零** → 见 §9.6 两种语义

### 9.5 显示控制按钮组（向大屏推送）

- **比赛视图 / 出发表 / 组成绩 / 总排名 / 团体积分 / 颁奖典礼 / 纪录榜 / 赛程表 / 晋级名单 / 欢迎画面 / 暂停画面**
- **成绩发布**：选择已完赛的组次，发布到大屏（独立锁定模式，不被实时画面覆盖）
- **图片 / 视频**：上传图片或视频全屏播放

### 9.6 计时器清零的两种语义

按"当前组是否已确认成绩"自动选择行为：

#### A) 已确认成绩的组（赛程导航已显示 [已完赛]）

- **彻底清零复位**，准备进入下一组比赛
- **已确认成绩永久保留**（在 `_confirmedHeats` + 各运动员 `Results` 中已落盘）
- 自动 `SetCurrentHeat(N+1)` 切到下一组
- 若已是本赛次最后一组：泳道行整体清空（"（空泳道）"），顶部"组别"行改为"第 N 组已完赛 / 共 M 组"，**大屏切到欢迎画面**

#### B) 未确认成绩的组（如抢跳召回重赛）

- 只清当前组的时间 / 分段数据
- **DSQ / DNS / DNF 等备注状态保留**
- 停留在当前组，重新发令

### 9.7 切组守卫

当前组未确认成绩时，禁止切换到其它项目 / 组次（保护操作员避免误操作丢成绩）。三个计时控制台（主控 / EXE / HTML）规则一致。

### 9.8 已确认组防再次开赛

`就位 / READY` 按钮：当前组已确认时，弹"操作被阻止"提示，必须切到下一组才能继续。

### 9.9 起跳犯规处理（人工判罚 + DSQ 道继续比赛）

| 步骤 | 系统行为 |
|---|---|
| 硬件检测到反应时 < `抢跳判定阈值` | 仅置 `IsSuspectFalseStart=true`，反应时数字变红、行边框 amber 提示 |
| 三端日志 | 输出 `⚠ 起跳可疑（待裁判确认）泳道X 反应时:0.082s` |
| 裁判判罚 | 在该泳道按 **DSQ** 按钮（三端均可）；备注栏立即显示 `DSQ`、运动员状态置 DSQ |
| 裁判不判罚 | 反应时仍标红，但成绩照常计入；后续可继续比赛 |
| **DSQ 道继续接收原始数据**（v2026.05.20）| 标 DSQ 后泳道仍处于"接收态"——触板/盲表/分段/反应时全部继续记录并落盘到原始数据；倒计时结束后下一段触板/盲表/出发台照常打开。仅大屏排名/最终成绩排除该道 |
| 实时上大屏 | 任何 DSQ 标注通过 `MARK_DSQ` 立刻广播到 `display.html`，备注栏立即出现 `DSQ`（不再等待"确认本组成绩"）|

> 实现：`IsLaneReceivingData(lane)` 用于"原始数据接收 + 设备开关"判定（DSQ 通过），`IsLaneParticipating(lane)` 仍用于"排名 / 积分"判定（DSQ 排除）。两者职责分离。

### 9.10 接力 4 棒反应时（v2026.05.20）

每棒交接时，服务器按棒次索引覆盖写入 `LaneResult.LegReactionTimes[legIdx]`：

- **第 1 棒反应时** = 出发台时间（相对发令信号）
- **第 N+1 棒反应时** = 第 N+1 棒出发台时间 − 第 N 棒触板累计时间（N=1, 2, 3）
- 由 `ComputeRelayLegIndex(currentLap)` 推算棒次索引，按索引"覆盖写入"而不是"追加"，硬件事件重复或乱序也只保留每棒最新值
- 同步广播 `sw.reactionSeq`（`LaneDeviceState.ReactionTime` setter 每次自增），客户端用它检测"新一棒反应时"，重启大屏备注栏 5 秒倒计时窗口
- 个人项目仍是 `StartingBlockTime` 单值

### 10.3 并列名次

`UpdateHeatRanking` / `ComputeLiveRanks` / `GetEventRanking` 三处排名计算统一使用 **competition ranking 1-2-2-4** 规则：

- 同 `FinalTime` 的运动员**共享名次**
- 下一位名次按已遍历个数跳过

例：A=54.50（1）、B=54.68（2）、C=54.68（2）、D=54.99（4）。

打印的每组成绩表中，并列名次以 `=2` 形式标注（首条为 `2`，第二个并列写 `=2`），保留可读性。

> **判定精度**（v2026.05.30）：硬件计时控制器以 1/100 秒精度（单字节）发送成绩。软件侧把成绩按 `Math.Round(t × 100, MidpointRounding.AwayFromZero)` 取长整数（百分秒）后做相等比较：两条都 round 到同一个百分秒整数即视为并列。这样和硬件精度完全对齐，避免 IEEE 754 浮点尾差（不同计时源裁定后产生的几 µs 差异）漏判并列。
> 5 处排名计算（`UpdateHeatRanking` / `GetEventRanking` / `ComputeLiveRanks` / 两处打印）+ `EnforceOnlyLeaderRecordNote` 全部走同一辅助 `IsTieTime`。

### 11.3 自动比对工作流

```
触板完赛  → CheckRecords() → 写 result.RecordNote
                          → EnforceOnlyLeaderRecordNote() → 仅本组第 1 名（含 5ms 内并列）保留
                                    ↓ 广播 + UI/打印 显示
                                    ↓ 裁判 ConfirmResult
                              UpdateRecordsAfterConfirm() → 刷新 SwimmingRecord
                                    ↓ AutoSaveData
                              入档持久化（CompetitionPackage._records）
```

| 阶段 | 行为 |
|---|---|
| **完赛瞬间** | `CheckRecords` 遍历所有匹配 `项目+性别+组别(可空)` 且 `Time>0` 的记录条目；`FinalTime < record.Time` 写破纪录、`FinalTime == record.Time` 写平纪录；记录库为空则不比较 |
| **本组只保留第 1 名**（v2026.05.20）| `EnforceOnlyLeaderRecordNote` 在 `CheckRecords` 末尾及 `UpdateHeatRanking` 末尾运行：取本组最快有效成绩 `leaderTime`，对所有 `FinalTime > leaderTime + 5ms` 的运动员 `RecordNote = ""`。本组多人破同一记录时，仅本组最快者（含并列）保留标识，符合 FINA 惯例 |
| **同步显示** | `result.RecordNote` 通过 `Broadcast` 同时推到主控泳道行（备注栏金色）、EXE 远程台、`race_control.html`、`display.html`（备注栏金色 `#FBBF24` 显示标识）|
| **打印输出** | 每组成绩打印模板的成绩单元格后追加 `<span style='color:#b45309;font-weight:bold;'>WR</span>`（FINA 惯例）；"成绩与排名" Tab 的"纪录"列同步填值 |
| **裁判确认** | 按 **确认本组成绩**，`UpdateRecordsAfterConfirm` 把仍 `< record.Time` 的成绩写回 `SwimmingRecord`：刷新 `HolderName / HolderCountry / Time / TimeInSeconds / Date(今天) / Location(_competitionName)`，落盘到 CompetitionPackage |
| **平纪录不刷新** | `=` 前缀（平纪录）只标识，不更新原条目 |
| **DSQ 取消标识** | 裁判把第 1 名标 DSQ 后，`UpdateHeatRanking` 重算时新晋第 1 名自动获得标识，原第 1 名（已 DSQ）的标识被清空 |

### 11.4 破纪录统计表

### 13.4 大屏分区刷新

为避免高频整屏 innerHTML 重建造成的卡顿：

- **顶部静态文字**（赛事名 / 项目 / 阶段·组 / 纪录条）：仅在数据变化时写入 textContent
- **滚动时间右上角**：100 ms 独立 ticker 本地插值，与网络广播解耦
- **泳道行 DOM**：仅在 `buildKey`（项目 / 组 / 泳道布局 / 接力切换）变化时重建一次，缓存到 `_raceRowDOM[]`
- **泳道字段**（道次 / 姓名 / 队伍 / 成绩 / 名次 / 备注）：250 ms 廉价刷新 + 收到广播时刷新；字段对比，仅写差异
- **滚动时间在确认成绩后清空**（v2026.05.20）：`_updateRunningTimeOnly` 检测 `data.resultConfirmed === true` 时直接清空 `#runningTime` 文本并 return，不被 ticker 覆盖回去

效果：滚动时间走时丝滑，泳道字段无重绘抖动；确认成绩后切到本组成绩榜，右上角不再有滚动时间干扰。

- **直播视图**：`_activeSwimmersForDisplay` 滤除 `status==='EMPTY'` 或无姓名+无号码的占位行（v2026.05.20）
- **成绩榜 / 公布成绩**：`isOccupiedLane` 同样过滤，0 道无人时不再显示"0"那一行

### 13.6 隐私保护

服务器广播载荷 v2026.05.20 起**取消** `notes` 字段。`Swimmer.Notes` 可能含运动员个人备注、姓名、单位、教练等敏感文本，不应在大屏/排名屏暴露。报名端编辑 `notes` 走独立的 `REGISTER_QUERY_RESULT` 通道。

### 13.7 图片 / 视频全屏

支持 PNG/JPG/MP4 全屏推送到大屏，三种贴合方式（contain / cover / fill）。视频可循环、静音、自动播放。

## 15. 参数设置与硬件连接

### 15.1 参数设置（与所有客户端 + 硬件双向同步）

| 参数 | 默认 | 说明 |
|---|---|---|
| 泳道关闭时间（秒）| 20 | 触板触碰后多久关闭其它设备 |
| 出发台关闭延迟 | 3 | 发令后延迟多少秒关出发台 |
| 成绩确认关闭延迟 | 3 | 确认成绩后延迟关闭出发台 |
| 抢跳判定阈值 | 0.10 | 反应时低于此值视为"可疑抢跳"（仅标红，不自动 DSQ）|
| 分段成绩显示时长 | 5 | 大屏显示分段时间的秒数 — 也是反应时显示窗口时长 |
| 第 1 名成绩停留时间 | 3 | 右上角"第 1 名分段成绩"绿色提示秒数 |

| 终点位置 / 发令位置 | 左 / 右 | 单选 |
| 左 / 右盲表数量 | 3 / 3 | 1-3 个 |

修改后立即广播到所有客户端 + 硬件，三个计时控制端（主控 / EXE / HTML）的"参数设置"对话框都内置上述所有条目。

> 主控参数设置弹窗为 `SizeToContent.Height` 自适应高度，不会因新增条目而显示不全。

#### 15.1.1 反应时(RT) 开 / 关详解

- **打开（默认）**：硬件检测到出发台触发后写入反应时；反应时低于阈值时自动标红"可疑起跳"（不再自动 DSQ）。
- **关闭**：所有反应时相关处理统一跳过——
  - `HandleFalseStartDetected` 直接 return
  - 出发台命中处理（普通+接力交接）：不写 `ReactionTime`、不判定可疑、不再标红
  - `result.StartingBlockTime` 不写入；接力 `LegReactionTimes` 不更新
  - 三端泳道行 `ReactionText` 与计时源详情面板：显示空 / "已关闭"
  - 广播 `reactionTime` 字段为空字符串

适用场景：训练记录 / 没有出发台硬件 / 不需要反应时统计的赛事。

#### 15.1.2 大屏翻屏时间详解

- 影响 `display.html` 总排名（多页超过 10 行时）的自动翻页周期
- 不影响其它显示模式（比赛视图 / 出发表 / 颁奖等）
- 实时生效：服务器广播后大屏立即采用新值，无需刷新页面

### 15.2 硬件连接

- 协议类型：串口 / UDP（局域网）/ TCP
- 串口参数：波特率（默认 9600）/ 串口号
- UDP / TCP：IP + 端口
- 测试 / 连接 / 断开 / 重连

## 23. 常见问题

### 23.1 主服务器闪退或无法启动

- 查看是否被防病毒软件拦截
- 检查 .NET Framework 4.5+ 是否安装
- 端口 3002 是否被其它进程占用

### 23.2 客户端连不上主服务器

- 检查 IP 是否正确（用 `ipconfig` 确认主机 IP）
- 防火墙是否放行 `SwimmingScoreboard.exe`
- 主服务器右下角"连接状态"是否显示已启动 WebSocket
- 浏览器端启动遮罩里的 IP 输入框可重输；按 3 次失败后会弹 confirm 询问是否换地址

### 23.3 大屏卡顿

- 已优化为分区刷新（100ms 滚动时间 + 250ms 廉价字段刷新 + DOM 缓存）
- 老版本浏览器（IE / 旧 Edge）渲染慢，建议 Chrome / 新 Edge

### 23.4 接力大屏看不到反应时

- 确认主服务器 EXE 是 v2026.05.20+ 版本（旧版没有 `reactionSeq` 字段，新版客户端会自动退回字符串比对，但旧 EXE 也可能有其它路径问题）
- 浏览器 Ctrl+F5 强制刷新（绕过缓存）
- 备注栏的反应时窗口由 `splitDisplayTime` 控制（默认 5 秒），过了就消隐；接力每棒交接时会重启窗口

### 23.5 接力打印反应时少于 4 棒

- 确认主服务器 EXE 是 v2026.05.20+ 版本（旧版按"追加"写 `LegReactionTimes`，硬件事件丢失/乱序时条目数会错；新版按棒次索引覆盖写入，槽位数固定 = 项目名 N×D 的 N）
- 未触发硬件事件的棒次显示 `—`（说明那棒确实没收到出发台事件）

### 23.6 标 DSQ 后该道触板/盲表不工作

- 确认 EXE 是 v2026.05.20+。旧版本 DSQ 后整道关闭，所有数据丢弃；新版仅排名/最终成绩排除 DSQ，原始数据继续接收

### 23.7 一组多人破纪录但大屏只显示第 1 名的标识

- 这是 v2026.05.20 起的 FINA 标准行为（`EnforceOnlyLeaderRecordNote`）。第 1 名（含 5ms 内并列）保留 RecordNote，其余清空。如需所有人都标，可在 `MainWindow.xaml.cs` 关掉这条调用

### 23.8 自动分组结果不符合 FINA 规则

- 请确认"项目类型"识别正确（短距 50/100/200 / 中长距离 400+ / 接力）
- 重新点 **预赛自动分组**（旧版数据手动调整后再点会重新执行 SW 3 算法）

### 23.9 已确认成绩消失

- 检查"赛程导航"是否仍显示 `[已完赛]`
- 在"成绩与排名"Tab 检查组次下拉是否包含该组
- 查看主控日志窗口是否有"赛程导航刷新失败 / 成绩与排名刷新失败"红色错误

### 23.10 接力队员重复登记

- 已修复：相同 (队名 + 性别 + 项目) 的接力队会被识别为更新，不会增加重复行

### 23.11 计时器清零后大屏没切到欢迎画面

- 仅在"本赛次最后一组已确认 + 计时器清零"才会切欢迎画面
- 中间组的清零会自动切到下一组（仍显示比赛视图）
