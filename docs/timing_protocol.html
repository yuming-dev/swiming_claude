游泳计时系统 通讯协议

主服务器（SwimmingScoreboard） ↔ 硬件计时控制器
版本 v2026-06  |  与参考程序 SWIM 串口通讯程序 Server 对齐  |  默认串口 9600/115200 8N1

1  文档目的

本文描述游泳计时系统主服务器（SwimmingScoreboard.exe）与硬件计时控制器之间的 串口 / UDP / TCP 帧级通讯协议。本版（v2026-06）相对 v2026-04 做了一次大幅对齐： 设置类命令一律改为参考程序 C:\2026年11月2日PM SWIM串口通讯程序Server 的命令编号 和参数布局（SwimView.h / OnSendSWCommand_Data）；原先用 0x42 + 子码 的方式硬件不识别，已废弃。本文档与 SwimmingScoreboard/TimingBridge.cs、 MainWindow.SendTimingSettingsToHardware、 SendDeviceStatusesToHardware、 HandleTimingCommand 的当前实现保持一致。

2  物理层与帧结构

2.1  物理层

项参数
链路串口（推荐）/ UDP / TCP 任选其一
串口参数9600 baud, 8 data bits, no parity, 1 stop bit（9600 n 8 1）
主服务器界面可改 9600 / 19200 / 38400 / 57600 / 115200，落盘到 timing_connection.json。
字节序按字节顺序发送，无大小端差异；所有参数值为 8 位无符号整数
帧定界起始字节 0xF1，结束字节 0xF4
帧长固定 12 字节

2.2  帧结构

位置字段值 / 含义说明
D0SOH0xF1起始标识（固定）
D1'S'0x53协议识别字符（固定）
D2CMD命令字节见第 3 章"命令表"
D3参数 1子码 / 第一参数计时帧：保留 0；设置帧：含义随命令
D4参数 2 / 泳道泳道号或参数值计时帧：0-9 终点端泳道，10-19 另一端泳道（实际泳道 = D4-10）
设置帧：参数值
D5分计时分计时帧有效；设置帧作扩展字段
D6秒计时秒计时帧有效；设置帧作扩展字段
D71/100 秒百分之一秒计时帧有效
D8小时|毫秒(hour << 4) | ms1高 4 位 = 小时，低 4 位 = 1/1000 秒个位
D9扩展 10 / 参数v2026-04 之前固定为 0；v2026-06 起 0x41 等命令使用
D10扩展 20 / 参数同 D9
D11EOT0xF4结束标识（固定）
解析要点：接收端应扫描起始字节 0xF1，检查 D1 = 0x53 且 D11 = 0xF4，不满足则丢弃该字节继续扫描；支持任意碎片累积到 12 字节后进行一次帧解析。
v2026-06 重要变化：D9 / D10 之前固定为 0（由 TimingBridge.SendFullFrame 保证）。新版 0x41 Set_ArmDelay_Time 帧把这两个位置当作 TPDelayTime / SBDelayTime 使用。SendFullFrame 已扩展到接受 d9 / d10 参数，旧版调用没传 d9 / d10 仍会写 0，不影响 0x40 / 0x44 / 0x45 / 0x46 等帧。

2.3  时间字段编码（计时帧）

以 "1 小时 23 分 45.678 秒" 为例（01:23:45.67 加 8 ms）：

字段D5 分D6 秒D7 1/100 秒D8 (小时<<4) | ms1
值 0x17 (23) 0x2D (45) 0x43 (67) 0x18 = (1<<4) | 8

主服务器解码公式：time = hour·3600 + min·60 + sec + cs/100 + ms1/1000

CMD名称方向含义 & 参数
0x16触板时间成绩硬件 → 主服D4 = 泳道号（0-9 终点端；10-19 另一端 - 10）
0x17盲表 1 时间成绩硬件 → 主服D4 同上
0x18盲表 2 时间成绩硬件 → 主服D4 同上
0x19盲表 3 时间成绩硬件 → 主服D4 同上
0x1A出发台出发时间硬件 → 主服D4 同上；用于反应时 / 接力抢跳检测
0x1C发令开始计时双向无参数
0x1D设备测试 (Test_Command)主服 → 硬件无参数。硬件进入测试模式：所有触板/出发台/盲表都允许触发并立刻上报，软件相应进入 _testMode（不计成绩）。参考程序 OnBnClickedTestSystemButton。
0x20计时清零双向无参数
0x21准备就绪双向无参数；通常在 0x43 Set_MatchEvent 之后发送（见 5.1）
0x7F滚动时间硬件 → 主服实时时钟广播，约 100ms 一帧；主服务器以此为权威时钟
0x10 0x42Set_SwStartingPosition主服 → 硬件D2=0x10（TCPIP_Con_command），D3=0x42（Set_SwStartingPosition），D4=发令端（0=左/1=右）
0x40设置泳池参数（旧）兼容接收v2026-04 旧版的泳池参数下发命令；v2026-06 改用 0x44。仍接收硬件可能上报的 0x40。
0x41Set_ArmDelay_Time主服 → 硬件触板/盲表/出发台 各项延迟一帧打包，使用全部 8 个数据字节（D3..D10）。
详见 4.1。
0x42Set_SwStartingPosition（独立）主服 → 硬件串口直接独立帧时使用：D3=发令端 (0/1)。
v2026-04 的 "0x42 + 子码 0x01..0x09" 用法已废弃。
0x43Set_MatchEvent主服 → 硬件当前组比赛参数：总圈数、左右触板预期次数、缺道位图。详见 4.2。
0x44Set_PoolConfiguration_Com1主服 → 硬件泳池参数：D3=泳道数 (8-10)，D4=泳池长度 (25/50)。替代旧 0x40。
0x45Set_MB_Num主服 → 硬件盲表数量：D3=左端数量 (1-3)，D4=右端数量 (1-3)。
0x46Set_TPSBMB_State主服 → 硬件触板/出发台/盲表 1 启用位流（一帧含全部 20 道左右两端）。详见 4.3。
替代旧 0x42 / 0x10 设备状态子码。
0x47Set_StartPlaceLaneDir主服 → 硬件发令端 + 道次方向位（与 0x41 d8 同一字节布局）。一般通过 0x41 一起下发，0x47 备用。
0x48Set_OpenOrClose_ALL_TP主服 → 硬件整体开/关全部触板（参考程序 OnSendSWCommand_Data(Set_OpenOrClose_ALL_TP, ..)）；当前主服未发，预留。
0x49盲表 2 启用位（扩展）主服 → 硬件v2026-06 新增。同 0x46 MB 段格式，3 字节 × 20 位编排。当前固件未识别会忽略，待硬件升级。
0x4A盲表 3 启用位（扩展）主服 → 硬件同 0x49，针对盲表 3。
命令编号约定（参考程序 SwimView.h）：所有 Set_* 命令以 0x30 + 序号 + 0x10 计算： Con_Display_SwDirection=0x40, Set_ArmDelay_Time=0x41, Set_SwStartingPosition=0x42, Set_MatchEvent=0x43, Set_PoolConfiguration_Com1=0x44, Set_MB_Num=0x45, Set_TPSBMB_State=0x46, Set_StartPlaceLaneDir=0x47, Set_OpenOrClose_ALL_TP=0x48。

4  设置类帧详细布局

4.1  0x41 Set_ArmDelay_Time（核心配置帧）

主服务器"参数设置"对话框点击确定后，所有触板/盲表/出发台的延迟、显示时长、发令端、道次方向都打包在这一帧里：

字段含义单位说明
D2 = 0x41命令—Set_ArmDelay_Time
D3ArmDelayNormalTime秒触板正常关闭时间。当前实现 = LaneCloseTime（默认 20）
D4ArmDelayAfterStartTime秒抢跳后触板关闭时间。当前实现 = LaneCloseTime（暂复用同值）
D5MBDelayTime秒盲表等待时间（无 TP 时盲表数据等待此时长后落地）。默认 50（参考程序值）
D6Result_Display_Time秒分段成绩显示时长，= SplitDisplayTime（默认 5）
D7StartboxEdgeBit0/1出发信号边沿：0 = 下降沿，1 = 上升沿。默认 0
D8方向 / 50m / 发令端 三合一位
bit 4bit 1bit 0
LaneDirectBit (1=逆序9→0 / 0=正序0→9) StartPlace50MBit (1=50米发令端不同于终点) StartPlaceBit (0=左 / 1=右)
即 D8 = LaneDirectBit*16 + StartPlace50MBit*2 + StartPlaceBit*1，与参考程序一致。
D9TPDelayTime秒触板关闭延迟，= ResultConfirmCloseDelay（默认 3）
D10SBDelayTime秒出发台关闭延迟，= StartBlockCloseDelay（默认 3）

例：默认参数 + 终点在左端 + 道次正序： D8 = 0×16 + 0×2 + 0×1 = 0x00。 若改逆序：D8 = 1×16 + 0 + 0 = 0x10。 若 50 米项目从右端发令：D8 = 0×16 + 1×2 + 1×1 = 0x03。

4.2  0x43 Set_MatchEvent（当前组）

主服务器选定项目 / 切换组次时下发，告诉硬件本组比赛的总圈数、左右预期触板次数、哪些泳道有运动员：

字段含义说明
D2 = 0x43命令Set_MatchEvent
D3总圈数 (All_Lap)例 100 米 / 50 米池 = 2；400 米 / 25 米池 = 16
D4RightExpected_Lap右端预期触板次数
D5LeftExpected_Lap左端预期触板次数
D6LaneOpen0_4 位图bit i (i=0..4) = 1 表示第 i 道有运动员（v2026-06 起统一为"有=1"）
D7LaneOpen5_9 位图bit i (i=0..4) = 1 表示第 (i+5) 道有运动员
触发时机：
  • 选项目 / 切组次（SetCurrentEvent / SetCurrentHeat）
  • "计时复位"按下后（硬件 0x20 会清掉缺道位图，需要重发）
  • 就位前的全量参数同步
缺道位图位序：bit 0..9 = 道 0..9；位 = 1 表示该道有运动员，位 = 0 表示该道空（v2026-04 b 版位序与含义已统一为此版本）。

4.3  0x46 Set_TPSBMB_State（设备启用位）

"设备状态管理"对话框确定后下发；包含触板 + 出发台 + 盲表 1 三大类、左右两端共 20 个泳道、共 60 个启用位。盲表 2 / 盲表 3 通过 0x49 / 0x4A 扩展帧独立发送。

4.3.1  20 位编排

每一类（TP / SB / MB）有 20 位：bit 0..9 = 左端 0..9 道，bit 10..19 = 右端 0..9 道。位 = 1 表示该位置启用（设备正常工作）；位 = 0 表示未用 / 损坏。

4.3.2  0x46 帧布局

字段含义编码
D2 = 0x46命令Set_TPSBMB_State
D3State[0]TP_Value bit 0..7
D4State[1]TP_Value bit 8..15
D5State[2]TP_Value bit 16..19（仅低 4 位有效）
D6State[3]SB_Value bit 0..7
D7State[4]SB_Value bit 8..15
D8State[5]MB1_Value bit 0..7
D9State[6]MB1_Value bit 8..15
D10State[7]高 4 位 = SB_Value bit 16..19；低 4 位 = MB1_Value bit 16..19
与参考程序 BadTP_SB_MBProg.cpp（OnBnClickedOk）的位流构造完全一致。每按下一次"设备状态管理 → 确定"主服务器一次性下发整帧；硬件以收到的 60 位为权威全量替换内部状态。

4.3.3  0x49 / 0x4A 扩展帧（盲表 2 / 盲表 3）

参考程序硬件每端每道只有 1 个 MB 槽（即上面 0x46 的 MB1）。本系统每端每道支持 3 只盲表，所以扩展两条命令把盲表 2 / 盲表 3 的启用位单独下发：

字段含义编码
D2命令0x49 = 盲表 2，0x4A = 盲表 3
D3MB2/3 bit 0..7同 0x46 MB 段
D4MB2/3 bit 8..15同上
D5MB2/3 bit 16..19仅低 4 位有效
D6..D10保留0
当前固件未识别 0x49 / 0x4A：主服务器一直发，固件忽略不报错。待硬件方升级到识别这两条后即可显式利用 3 只盲表的位图，主服务器侧无需再修改。

4.4  0x44 Set_PoolConfiguration_Com1（泳池参数）

字段含义说明
D2 = 0x44命令Set_PoolConfiguration_Com1
D3泳道数6 / 8 / 10
D4泳池长度25 / 50（米）

4.5  0x45 Set_MB_Num（盲表数量）

字段含义说明
D2 = 0x45命令Set_MB_Num
D3左端盲表数量1 / 2 / 3
D4右端盲表数量1 / 2 / 3

4.6  0x10 0x42 Set_SwStartingPosition（发令端）

这是 TCPIP 包装版本（TCPIP_Con_command=0x10 + Set_SwStartingPosition=0x42）。串口直连场景下，直接发 F1 53 42 [pos] 00 00 00 00 00 00 00 F4 即可（D2=0x42, D3=发令端）。

5  运行流程

5.1  就位前的全量同步（硬件刚连上 / Ready 之前）

0x44 D3=Lanes  D4=PoolLen                                                  // 泳池
0x41 D3..D10 = ArmNorm/ArmAfter/MBDelay/ResultDisp/Edge/Place|Dir/TPDelay/SBDelay
0x45 D3=LeftMBNum  D4=RightMBNum                                            // 盲表数量
0x46 D3..D10 = TP/SB/MB1 启用位流                                            // 设备状态
0x49 D3..D5 = MB2 启用位流                                                   // 盲表 2 扩展
0x4A D3..D5 = MB3 启用位流                                                   // 盲表 3 扩展
0x43 D3=All_Lap D4=RightExp D5=LeftExp D6=LaneOpen0-4 D7=LaneOpen5-9         // 当前组
0x21                                                                         // 准备就绪

每帧之间 20-50ms 间隔（TimingBridge.DelayBetweenFrames），低波特率硬件需要时间消化。

5.2  比赛过程（硬件 → 主服务器）

事件命令主服务器处理
滚动时间0x7F更新 _runningTime + 立即广播 RUNNING_TIME_UPDATE
触板0x16校验该端是 Open（含 FalseStart）→ 写分段；切到 Touched（红）保持 ResultConfirmCloseDelay 秒；期间额外触板写 BackupTouchTimes
盲表 1/2/30x17/0x18/0x19逐只盲表独立校验 + 写分段；同 Touched 流程，期间写 BackupBlindTimes
出发台0x1A写反应时（接力下为 currentTouch − lastTouch）；该端切 Touched 保持 StartBlockCloseDelay 秒，期间写 BackupReactionTimes
硬件按键 就位0x21调 Ready_Click(null,null) 切到 Ready 状态
硬件按键 发令0x1CStartRace_Click → 进入 Racing
硬件按键 复位0x20Restart_Click，并补发 0x7F + 0x43（防止 0x20 清掉缺道位图）

5.3  复位后的二次同步

硬件 0x20 会顺带清掉它内部的缺道位图。主服务器无论是本地按 计时复位 还是收到硬件按键还是 HTML/EXE 远端命令，都按以下顺序补帧让硬件恢复："本组缺道"信息：

0x20    // 计时清零
0x7F    // 滚动时间归零
0x43    // 重发 Set_MatchEvent（含本组缺道位图）

5.4  设备测试模式（v2026-06 新增）

主服务器/HTML/EXE 任意一端按下 设备测试 按钮：

  1. 本端弹确认框（远端通过 WebSocket DEVICE_TEST_TOGGLE 请求服务器，服务器再切换）
  2. 主服务器 _testMode = true；清空 _rawTimingLog；所有非损坏/未安装设备置 Open
  3. 下发 0x1D Test_Command 给硬件 → 硬件进入测试模式：所有 TP/SB/MB 触发都允许并直接上报
  4. 硬件来事件 → 跳过比赛状态/空泳道过滤 → 写日志"【测试】..."；该端设备点 红闪 500ms 后回绿；姓名栏显示左端事件文字（"触 23.45"），成绩栏显示右端事件文字
  5. 三端状态徽章覆盖红色"设备测试"
  6. 测试期间不创建 split / 不计成绩 / 不进 Touched 窗口
  7. 再点一次按钮退出，所有设备 Closed，可恢复正常比赛
