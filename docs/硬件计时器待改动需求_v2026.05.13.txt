┌──────────────────────────────────────────────────────────────┐
│ 1. 【必修】0x1A StartingBlock 抢跳符号位                      │
└──────────────────────────────────────────────────────────────┘

  现状
  ────
  硬件出发台触发上行 0x1A 帧时，D10（SW_Back1_Pos，第 11 字节）
  始终为 0。PC 端按 D10 ≠ 0 判抢跳并把反应时取负显示，因此 PC
  永远收不到正确的抢跳标志，反应时永远显示为正值。

  修改
  ────
  代码位置：APP/swimplay.c 约 3588 行，发送 0x1A 帧的位置。
  在抢跳判定分支里给 D10 赋非零值：

      TXD_Buffer[SW_Back1_Pos] = 1;   // 0x01 即可，PC 仅判非 0

  时机：当出发台离台时刻早于发令枪（即检测出运动员抢跳/犯规）。
        若抢跳/正常判别已在硬件做了，就在该分支里加一行。
        如果是按"运动员出发时刻 - 发令枪时刻"得到的差值符号判断：
            if(差值 < 0)  TXD_Buffer[SW_Back1_Pos] = 1;
            else          TXD_Buffer[SW_Back1_Pos] = 0;

  PC 端配合（已就位）
  ──────────────────
  TimingBridge.cs:340 已实现：
      bool isFalseStart = false;
      if (cmd == 0x1A && rawD10 != 0) {
          isFalseStart = true;
          timeInSeconds = -timeInSeconds;
      }


┌──────────────────────────────────────────────────────────────┐
│ 2. 【必修】0x4B BatteryVoltage 电池电压周期上报              │
└──────────────────────────────────────────────────────────────┘

  现状
  ────
  APP/swimplay.c 第 1733 行注释提及，但完全没有实现 ADC 采样和
  发送代码。PC 端顶部状态栏的电压显示永远为空。

  协议规约
  ─────────
  方向：硬件 → PC（硬件主动周期上报）
  命令字节 D2 = 0x4B（直接发送，无 +0x10 偏移）
  帧格式（大端）：
      D2 = 0x4B
      D3 = mV 高字节
      D4 = mV 低字节
      D5–D10 = 0
  硬件电池：7.4V 标称（2 节锂电串联包，单节 3.7V）。
            满充 8.4V / 标称 7.4V / 低 7.0V / 耗尽截止 6.0V。

  例：7.85 V = 7850 mV = 0x1EAA
      F1 53 4B 1E AA 00 00 00 00 00 00 F4

  USER/swimtime.h 已有定义：
      #define Report_Voltage_RAW    0x4B

  修改建议
  ─────────
  在 swimplay.c 里加一个定时任务（如每 5 秒一次，或在系统主循环
  里按时间戳判断），完成：
  ① 读取电池电压 ADC（单位 mV）
  ② 拼帧并通过现有发送函数下发到 PC

  伪代码：
      static u32 last_volt_send_tick = 0;
      if (HAL_GetTick() - last_volt_send_tick >= 5000) {
          last_volt_send_tick = HAL_GetTick();
          u16 mv = ReadBatteryMV();         // 用户已有的 ADC 接口
          u8 buf[12];
          buf[0] = SOH;            // 0xF1
          buf[1] = 'S';            // 0x53
          buf[2] = Report_Voltage_RAW;     // 0x4B  ← 直接写，不要 +0x10
          buf[3] = (mv >> 8) & 0xFF;       // mV 高字节
          buf[4] = mv & 0xFF;              // mV 低字节
          buf[5] = 0; buf[6] = 0; buf[7] = 0;
          buf[8] = 0; buf[9] = 0; buf[10] = 0;
          buf[11] = EOT;           // 0xF4
          TCPIP_Send(buf, 12);     // 或者你已有的统一发送函数
      }

  PC 端配合（已就位）
  ──────────────────
  TimingBridge.cs:360  case 0x4B 已就位
  MainWindow.xaml.cs   ProcessTimingDataFromHardware → BatteryVoltage 分支
      mv  = (D3 << 8) | D4
      V   = mv / 1000.0
      顶栏显示 + 阈值染色（按 7.4V 包 / 2 节锂电串联）：
          <  7.00 V          红 (严重低电，请立即更换/充电)
          7.00 V – 7.40 V    琥珀 (偏低，注意监视)
          >= 7.40 V          绿 (健康)


┌──────────────────────────────────────────────────────────────┐
│ 3. 【选修】0x42 SetCommand 新增子码 0x09                     │
└──────────────────────────────────────────────────────────────┘

  现状
  ────
  PC → 硬件 已通过 0x41 帧 D5 把"盲表代替成绩延迟"下发，硬件已
  接收（swimplay.c:5447 MBdelay_Time = RXD[5]*10）。主路径 OK。

  USER/swimtime.h 已有定义：
      #define SetCmd_Sub_BlindReplaceDelay    0x09

  缺口
  ────
  硬件想"主动通知 PC 该参数变了"时，需经 0x42 子码路径。当前
  swimplay.c case 0x32 的子码分发表没有 0x09 分支。

  修改（如果有"硬件 → PC 回报参数"需求）
  ──────────────────────────────────────
  case 0x32 的子码 switch 里加：
      case SetCmd_Sub_BlindReplaceDelay:    // 0x09
          // d4 = 0.1秒单位，例如 50 = 5.0s
          MBdelay_Time = RXD_Data_Buffer[4] * 10;
          break;

  若硬件没有"在 GUI 改了参数再回传 PC"的需求，本项可不做。
  PC 端 ApplyHardwareSettingFrame:2323 已就位接收。


┌──────────────────────────────────────────────────────────────┐
│ 4. 【背景信息】PC 端已修复内容（你不用做任何事）              │
└──────────────────────────────────────────────────────────────┘

  Bug：PC 之前发"盲表 2/3 扩展位图"用了 0x49 / 0x4A。
       硬件 dispatcher 把 D2 - 0x10 = 0x39 / 0x3A，分别撞到
       Set_DateTime / Set_PoolSingleOrDoubleTP 的 case，导致每
       次同步设备状态都误改硬件 RTC 或切到单端模式。

  PC 端修复：已注释 MainWindow.xaml.cs:2591-2602 两条
            SendFullFrame(0x49…) / SendFullFrame(0x4A…)。
            盲表 2/3 的损坏 / 未安装信息只在 PC 本机保留。

  若硬件以后扩展盲表 2/3 位图，请使用 ≥ 0x60 段的命令码
  （避开 dispatcher 的 0x30+N 段），届时 PC 端再启用对应发送。
